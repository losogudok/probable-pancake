# Спецификация тулов первичной онтологической разметки текста

## Стадия 1: Суть

Разрабатываем набор простых Python-тулов без LLM для первичной разметки технического текста сущностями онтологии.

Главная идея: GPT заранее разделяет на примерных текстах три разных слоя и готовит простой "механический" разборщик.

1. **Известные термины онтологии** — фрагмент найден в `ontology2.py` как `TermSpec` или alias. В разметке показывается конкретная сущность: `EntityType:canonical`, например `Metric:НДС` или `Parameter:глубина`.
2. **Параметрические значения известных или локально распознанных параметров** — фрагмент вида `глубина=10 м`, `диаметр=96 мм`, `напряжение=15 МПа`. Это не новый термин онтологии, а числовое упоминание параметра в источнике.
3. **Кандидаты на пополнение онтологии** — фрагмент похож на сущность какого-то типа, но ещё не описан в `ontology2.py`. В разметке показывается только тип: `Method`, `Process`, `Equipment`, `Model` и т.п.

Тулы не принимают окончательных онтологических решений. Они помогают быстро увидеть:

- что уже покрыто текущей онтологией;
- какие важные термины документа ещё не заведены;
- какие числовые параметры можно извлекать простыми правилами;
- какие фрагменты затем стоит подтвердить человеком и превратить в новые `TermSpec`.

Целевой сценарий: пользователь даёт Markdown/text-документ и файл онтологии `ontology2.py`. На выходе получает HTML-подсветку, CSV со всеми совпадениями, CSV известных терминов, CSV числовых параметров и CSV кандидатов на пополнение онтологии.

## Полный pipeline обработки, пример

1. Подготовить входной текстовый документ, например результат парсинга PDF в Markdown.
2. Подготовить онтологию `ontology2.py`.
3. При необходимости подготовить словарь единиц `vocabulary/units.json`.
4. Запустить инвентаризацию слов и фраз:
   - вход: `.md` / `.txt`;
   - выход: `unique_words_entity_guesses.csv`, `phrase_candidates.csv`.
5. Запустить онтологический подсветчик:
   - вход: документ, `ontology2.py`, результаты инвентаризации, `units.json`;
   - выход: HTML и CSV-разметка.
6. Открыть `highlighted_entities.html` и посмотреть разметку.
7. Открыть `ontology_gap_candidates.csv`, выбрать полезные кандидаты.
8. Добавить подтверждённые кандидаты в онтологию как `TermSpec` или в отдельный domain module.
9. Перезапустить подсветку и проверить рост покрытия.

Рабочий цикл не должен требовать LLM. LLM можно подключать позже только для semi-automatic review кандидатов, но базовый тул должен работать детерминированно.

### Что считать непроизводным

Непроизводные файлы, которые надо хранить и править руками:

- `ontology_entity_highlighter.py`
- `ontology2.py`
- `tools/ontology_word_inventory.py`
- `vocabulary/units.json`
- входные документы в `examples/` или в отдельном corpus-каталоге

Производные файлы, которые можно удалять и пересоздавать:

- `outputs/**`
- `*_matches.csv`
- `known_terms.csv`
- `parameter_values.csv`
- `ontology_gap_candidates.csv`
- `highlighted_entities.html`
- `entity_highlight_stats.json`
- `*.zip` с результатами

Комментарии в .py пиши по-русски.

## Тул 1: `ontology_word_inventory.py`

### Назначение

Вспомогательный тул для грубой инвентаризации документа. Он нужен не для загрузки в граф, а для подготовки кандидатов:

- уникальные нормализованные слова;
- поверхностные формы слов;
- частоты;
- примерные контексты;
- эвристические догадки `EntityType`;
- кандидаты в многословные термины.

Тул не использует `ontology2.py`. Он работает от самого текста и набора простых эвристик.

### Вход

Один текстовый файл:

- `.md`
- `.txt`
- другой plain-text файл после парсинга исходного документа

### Основные действия

1. Прочитать документ.
2. Удалить или ослабить Markdown-служебные элементы, image placeholders, лишнюю пунктуацию.
3. Извлечь токены.
4. Нормализовать токены:
   - регистр;
   - `ё/е`;
   - пробелы;
   - простые варианты дефисов.
5. Сгруппировать surface forms по normalized key.
6. Посчитать частоты.
7. Для каждого слова дать грубую эвристическую догадку типа сущности.
8. Сформировать n-граммные phrase candidates.
9. Записать CSV и краткую статистику.

### Выходы

`unique_words_entity_guesses.csv` — уникальные слова / токены:

| Поле | Смысл |
|---|---|
| `normalized` | Нормализованная форма, главный ключ. |
| `surface_forms` | Исходные варианты написания. |
| `count` | Частота в документе. |
| `entity_type_guess` | Эвристическая догадка типа: `Process`, `Equipment`, `Method`, ... |
| `confidence` | Условная уверенность эвристики. |
| `reason` | Почему поставлен такой тип. |
| `contexts` | Несколько коротких контекстов. |

`phrase_candidates.csv` — кандидаты в многословные сущности:

| Поле | Смысл |
|---|---|
| `phrase` | Фраза-кандидат. |
| `normalized` | Нормализованная фраза. |
| `count` | Частота. |
| `entity_type_guess` | Грубая догадка типа. |
| `confidence` | Условная уверенность. |
| `contexts` | Короткие контексты. |

`summary.json` / `report.md` — опциональная статистика.

### Ограничения

- Это не NER.
- Это не строгая онтологическая разметка.
- Одиночные слова часто шумные.
- Главная ценность — найти недостающие фразы и подготовить вход для второго тула.

## Тул 2: `ontology_entity_highlighter.py`

### Назначение

Основной тул разметки. Совмещает:

1. строгую подсветку известных терминов из `ontology2.py`;
2. извлечение parameter-value mentions по простым правилам;
3. подсветку эвристических кандидатов из `phrase_candidates.csv` и `unique_words_entity_guesses.csv`;
4. формирование gap-файла для пополнения онтологии.

Тул должен заменять старый `ontology_registry_highlighter.py`. Отдельный строгий подсветчик больше не нужен, потому что строгий режим является частью основного тула.

### Входы

Обязательные:

- входной `.md` / `.txt` документ;
- `--ontology vocabulary/ontology2.py`;
- `--out-dir outputs/<run_name>`.

Желательные:

- `--phrases outputs/<inventory_run>/phrase_candidates.csv`;
- `--words outputs/<inventory_run>/unique_words_entity_guesses.csv`;
- `--units vocabulary/units.json`.

Параметры:

| Параметр | Смысл |
|---|---|
| `--domains` | Домены онтологии для разметки. По умолчанию разумно `common,underground_mining` для НДС/геомеханики. |
| `--word-min-confidence` | Минимальная уверенность word-кандидатов. |
| `--phrase-min-count` | Минимальная частота phrase-кандидатов. |

### Основной принцип разметки

Если фрагмент найден в онтологии как `TermSpec` или alias, он размечается как конкретная сущность:

```text
Metric:НДС
Parameter:напряжение
RockMass:горный массив
GeologicalFeature:трещиноватость
```

Если фрагмент не найден в онтологии, но эвристически похож на сущность, он размечается только типом:

```text
Method
Process
Equipment
Model
Organization
```

Если фрагмент является значением параметра, он размечается как parameter-value mention:

```text
Parameter:глубина=10 м
Parameter:диаметр=96 мм
Parameter:напряжение=15 МПа
```

Важно: `Parameter:глубина=10 м` — не новый `TermSpec`. Это конкретное упоминание значения в источнике. В графе оно должно стать свойством факта, измерением или узлом/связью уровня evidence, но не canonical term.

### Приоритеты при пересечениях

Если несколько совпадений пересекаются, оставлять более полезное:

1. parameter-value mention;
2. известный термин онтологии;
3. более длинная фраза;
4. более высокая уверенность;
5. эвристический кандидат.

Пример: если найдено `глубина` и `на глубину 10 м`, оставить `Parameter:глубина=10 м`.

### Нормализация терминов

Использовать registry из `ontology2.py`:

- `TermSpec.id` — стабильный canonical id;
- `TermSpec.type` — entity type;
- `TermSpec.canonical` — canonical label;
- `TermSpec.aliases` — варианты написания;
- `TermSpec.domain` — домен;
- `TermSpec.match` — подсказка для политики matching.

Поддерживать простую русскую морфологическую терпимость для однословных терминов. Например, если в онтологии `глубина`, в тексте надо поймать `глубину`, `глубине`, `глубинах` хотя бы простыми регулярными окончаниями. Это не полноценная лемматизация, а инженерный компромисс.

### Parameter-value extraction

Тул должен ловить конструкции:

```text
<известный параметр> + <0..несколько промежуточных слов> + <число/диапазон> + <единица>
```

Примеры:

```text
на глубину 10 м
на глубинах скважины 9-12 м
глубина скважины 10 м
на глубине 750 м
на глубину 0,4 м
напряжения порядка 15 МПа
```

Также можно ловить локальные параметрические кандидаты, даже если параметр ещё не описан в онтологии:

```text
диаметром HQ (96 мм) → Parameter:диаметр=96 мм
длиной 300-400мм → Parameter:длина=300-400 мм
расстоянии 10 см → Parameter:расстояние=10 см
шагом в 5 см → Parameter:шаг=5 см
```

Для ontology-based values источник должен быть `ontology_value`. Для локальных rule-based значений — `unit_value_candidate` или аналогичный источник с меньшей confidence.

### Словарь единиц `units.json`

Единицы должны быть вынесены из кода в `vocabulary/units.json`.

Минимальная структура записи:

| Поле | Смысл |
|---|---|
| `id` | Стабильный id единицы. |
| `canonical` | Каноническое отображение: `м`, `мм`, `МПа`. |
| `aliases` | Варианты: `m`, `mm`, `mpa`, `мпа`. |
| `quantity` | Тип величины: `length`, `pressure`, `temperature`, `force`, ... |
| `requires_context` | Опционально: какой контекст нужен для неоднозначной единицы. |

`K/К` нельзя считать Кельвином всегда. Для текущего класса текстов `K/К` как единица температуры допустим только при температурном контексте. Обозначения вроде `кН` должны быть отдельной единицей силы, а не `К` + `Н`.

Словарь единиц может дополнять `UNIT_PATTERNS` из `ontology2.py`, но не должен полностью зависеть от них.

## HTML-разметка

Файл `highlighted_entities.html` — основной человекочитаемый результат.

Требования:

- исходный текст сохраняется в читаемом виде;
- Markdown может отображаться как plain text, без полноценного рендера;
- совпадения подсвечиваются цветом по `entity_type`;
- известные термины и parameter-value mentions визуально отличать от эвристических кандидатов;
- на hover/title показывать минимум:
  - `display_label`;
  - `source`;
  - `term_id`, если есть;
  - `value` и `unit`, если есть;
  - `confidence`;
- в шапке показать статистику:
  - всего совпадений;
  - известных терминов;
  - parameter-value mentions;
  - gap candidates;
  - распределение по типам.
Шапка не должна закрывать собой основной текст.

Рекомендуемая визуальная семантика:

- сплошное подчёркивание — известная сущность онтологии;
- усиленная/отдельная подсветка — parameter-value mention;
- пунктирное подчёркивание — эвристический кандидат;
- разные цвета — разные `EntityType`.

### Служебные маркеры не являются самостоятельными сущностями

Некоторые слова похожи на сущности, но в базовой разметке не должны становиться самостоятельными узлами графа. Они используются как маркеры для сборки более крупных сущностей.

Примеры:

- `ООО`, `АО`, `ПАО`, `Ltd`, `Inc` — маркеры организационно-правовой формы.
- `департамент`, `лаборатория`, `отдел` — маркеры подразделений.
- `докладчик`, `заведующий`, `начальник`, `директор` — маркеры роли или должности.
- `к.т.н.`, `д.т.н.`, `PhD` — маркеры учёной степени.
- `метод`, `система`, `установка`, `модель` — типовые маркеры, которые сами по себе не всегда являются сущностью.

Такие маркеры должны либо:
1. подавляться как отдельные совпадения;
2. входить в состав более длинной сущности;
3. сохраняться как атрибут собранной сущности;
4. использоваться для построения отношения.

## CSV-выходы основного тула

### `entity_matches.csv`

Главный отладочный файл. Содержит все совпадения после разрешения пересечений.

Поля:

| Поле | Смысл |
|---|---|
| `start` | Смещение начала в исходном тексте. |
| `end` | Смещение конца. |
| `surface` | Фрагмент из текста. |
| `display_label` | Метка для UI: `Metric:НДС`, `Parameter:глубина=10 м`, `Equipment`. |
| `specificity` | `specific`, `generic`, `value`. |
| `entity_type` | Тип сущности. |
| `source` | `ontology`, `ontology_value`, `heuristic_word`, `heuristic_phrase`, `unit_value_candidate`. |
| `canonical` | Canonical term, если есть. |
| `term_id` | `TermSpec.id`, если есть. |
| `domain` | Домен онтологии или `local_rules`. |

| `value` | Числовое значение, если есть. |
| `unit` | Каноническая единица, если есть. |
| `confidence` | Условная уверенность. |
| `count` | Частота кандидата, если применимо. |
| `context` | Короткий контекст. |

### `known_terms.csv`

Только известные термины онтологии без числовых значений.

Поля:

| Поле | Смысл |
|---|---|
| `term_id` | Canonical id из онтологии. |

### `parameter_values.csv`

Только числовые параметрические упоминания.

Поля:

| Поле | Смысл |
|---|---|
| `surface` | Исходный фрагмент. |
| `display_label` | Например `Parameter:глубина=10 м`. |
| `entity_type` | Обычно `Parameter`, иногда `Metric`/`Property`, если будет расширено. |
| `canonical` | Параметр: `глубина`, `диаметр`, `напряжение`. |
| `term_id` | Id известного термина или локальный id вроде `local:Diameter`. |
| `domain` | Домен или `local_rules`. |
| `value` | Нормализованное число или диапазон. |
| `unit` | Каноническая единица. |
| `source` | `ontology_value` или `unit_value_candidate`. |
| `confidence` | Уверенность. |
| `start`, `end` | Смещения в тексте. |
| `context` | Контекст. |

### `known_entities.csv`

Сводный файл для совместимости. Включает и известные термины, и parameter-value mentions.

Обязательное поле `entity_kind`:

- `term` — canonical term из ontology registry;
- `parameter_value` — числовое упоминание известного или локального параметра.

### `ontology_gap_candidates.csv`

Кандидаты на пополнение онтологии.

Сюда попадают только generic heuristic matches, которых нет в ontology registry.

Поля:

| Поле | Смысл |
|---|---|
| `surface` | Фрагмент. |
| `normalized` | Нормализованный ключ. |
| `entity_type_guess` | Предполагаемый тип. |
| `count` | Частота. |
| `confidence` | Уверенность. |
| `contexts` | Примеры. |
| `suggested_action` | Например `review_as_termspec`, если добавляется позже. |

## Статистика запуска

`entity_highlight_stats.json` должен содержать:

- путь входного файла;
- путь онтологии;
- domains;
- число загруженных ontology terms;
- число aliases;
- число загруженных units;
- число всех matches;
- число known terms;
- число parameter values;
- число gap candidates;
- распределение по `entity_type`;
- распределение по `source`.

## Рабочий процесс пополнения онтологии

1. Открыть `ontology_gap_candidates.csv`.
2. Отфильтровать частотные и доменно важные кандидаты.
3. Для каждого кандидата решить:
   - это реальный термин;
   - это форма/alias уже существующего термина;
   - это мусор;
   - это не термин, а параметрическое значение;
   - это лучше оставить локальным правилом.
4. Для реальных терминов добавить `TermSpec` в `ontology2.py` или отдельный domain module.
5. Для aliases добавить alias к существующему `TermSpec`.
6. Для единиц — поправить `vocabulary/units.json`.
7. Перезапустить оба тула.
8. Проверить, что кандидат перешёл из `ontology_gap_candidates.csv` в `known_terms.csv` или `parameter_values.csv`.

## Что не делать в этих тулах

- Не использовать LLM в базовом режиме.
- Не писать напрямую в Neo4j.
- Не считать эвристический кандидат фактом.
- Не создавать новые `EntityType` автоматически.
- Не считать parameter-value mention новым термином онтологии.
- Не делать глубокую морфологию, если хватает простых правил.
- Не смешивать непроизводные vocabulary-файлы и производные результаты.

## Минимальная проверка качества
На документе про НДС / overcoring тул должен, как минимум:

1. Найти известные термины из онтологии:
   - `Metric:НДС`;
   - `Parameter:напряжение`;
   - `GeologicalFeature:трещиноватость`;
   - `RockMass:горный массив`;
   - `Parameter:глубина`.
2. Найти parameter-value mentions:
   - `на глубину 10 м`;
   - `на глубинах скважины 9-12 м`;
   - `глубина скважины 10 м`;
   - `на глубине 750 м`;
   - `на глубину 0,4 м`;
   - `диаметром HQ (96 мм)`;
   - `диаметра 46 мм`;
   - `длиной 300-400мм`;
   - `расстоянии 10 см`;
   - `шагом в 5 см`;
   - `400 Па` и `15 МПа` как значения напряжения, если рядом есть контекст напряжений.
3. Не размечать `К.Т.Н.` как Kelvin.
4. Вынести отсутствующие, но важные термины в gap candidates:
   - `метод полной разгрузки керна`;
   - `Overcoring`;
   - `экстензометр`;
   - `скважинный деформометр`;
   - `тензодатчик`;
   - `USBM Cell`;
   - `IST tool`;
