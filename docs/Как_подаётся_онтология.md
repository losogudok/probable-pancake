# Как подаётся онтология

В проекте онтология подана в двух слоях: как краткий предметный контракт для людей и ИИ-агентов, и как исполняемый словарь, на котором держатся извлечение сущностей, построение графа и RAG.

## 1. Markdown-глоссарий

Главное человекочитаемое описание находится в [`CONTEXT.md`](../CONTEXT.md). Это компактный глоссарий без имплементационных деталей. Он фиксирует:

- canonical-сущности предметной области: `Material`, `Process`, `Equipment`, `Facility`, `Experiment`, `Publication`, `Patent`, `Expert`, `Property`, `Parameter`, `Condition`, `Recommendation`;
- canonical-связи: `uses_material`, `uses_reagent`, `uses_equipment`, `uses_method`, `performed_at`, `performed_by`, `validated_by`, `contradicted_by`, `requires`, `has_condition`, `has_parameter`, `has_result`, `derived_from`, `cites`;
- домены: гидрометаллургия, пирометаллургия, экология, переработка отходов;
- географию: отечественная практика, зарубежная практика, глобальная;
- модель верификации: источник, уровень достоверности, дата актуализации, противоречие, пробел знаний;
- RU/EN-синонимы, например `электроэкстракция = electrowinning`, `ПВП = fluidized bed furnace`, `штейн = matte`.

Этот файл можно читать как минимальный domain contract: он объясняет, какие объекты и отношения система считает допустимыми и как их называть.

## 2. Исполняемая онтология

Кодовая версия находится в [`app/core/ontology.py`](../app/core/ontology.py). В ней Markdown-идея превращена в структуры, которые реально используются пайплайном:

- `EntityType` задаёт типы узлов графа: материалы, процессы, оборудование, публикации, эксперты, параметры, методы, результаты, химические вещества и условия;
- `RelationType` задаёт типы рёбер, включая связи из глоссария и дополнительные рабочие отношения вроде `DESCRIBED_IN`, `IMPROVES`, `PRODUCES`, `APPLIES_TO`;
- `SourceType` и `TRUST_SCORES` задают шкалу доверия к источникам;
- `Geography` нормализует географическую классификацию;
- `SYNONYMS` и `normalize_term()` приводят RU/EN-алиасы к canonical-формам;
- keyword sets (`MATERIAL_KEYWORDS`, `PROCESS_KEYWORDS`, `EQUIPMENT_KEYWORDS`, `PARAMETER_KEYWORDS`) служат основой эвристического извлечения.

То есть онтология здесь работает не как OWL/RDF-схема, а как controlled vocabulary и набор правил нормализации для закрытого металлургического домена.

## 3. Как она используется в пайплайне

Сборка запускается через [`app/build.py`](../app/build.py). Документы проходят ingestion, разбиваются на чанки, после чего для каждого чанка вызывается extractor из [`app/nlp/extractor.py`](../app/nlp/extractor.py).

Фактический путь такой:

1. `ingestion.pipeline` парсит документы, определяет метаданные, язык, географию, год, тип источника и trust score.
2. `heuristic_extract_entities()` ищет в тексте ключевые слова из онтологии и создаёт сущности с типами `Material`, `Process`, `Equipment`, `Parameter`.
3. `extract_numeric_values()` regex-ом достаёт числовые значения с единицами и пытается привязать их к ближайшему параметру из ontology keywords.
4. `heuristic_extract_relations()` строит связи по совместному появлению в чанке: процесс использует материал, процесс использует оборудование, процесс имеет параметр.
5. При включённом `--use-llm` дополнительно запускается LLM-извлечение: модель получает список допустимых типов сущностей и связей и должна вернуть JSON.
6. [`app/graph/kg.py`](../app/graph/kg.py) добавляет сущности и связи в `networkx.MultiDiGraph`.

У каждого узла и ребра сохраняется provenance:

- `source_ids` — из каких документов пришёл факт;
- `trust` — максимальный уровень доверия среди источников;
- `geography` — география документа;
- `properties` — дополнительные данные, например `value`, `unit`, `evidence`.

## 4. Что делается с графом

Построенный граф используется как слой структурированных знаний поверх полнотекстового поиска:

- API отдаёт список сущностей и локальные окрестности узлов;
- RAG-движок в [`app/rag/engine.py`](../app/rag/engine.py) добавляет к найденным фрагментам факты из графа;
- `find_contradictions()` ищет числовые расхождения по одному процессу и параметру, если разброс больше 30%;
- `find_gaps()` ищет важные пары процесс-материал без связи;
- frontend визуализирует окрестности сущностей как граф.

На практике онтология поэтому выполняет три роли:

- ограничивает пространство допустимых сущностей и связей;
- нормализует разные написания и языки к canonical-формам;
- даёт структуру, по которой текстовые факты превращаются в графовые узлы, рёбра и проверяемые claims.

## 5. Важное расхождение в документации

В [`ARCHITECTURE.md`](../ARCHITECTURE.md) описан более тяжёлый целевой/боевой вариант: Neo4j, отдельные `grammar.py`, `gazetteer.py`, структурный OCR, артефакты `facts.jsonl`, `edges.jsonl`, `emb.npy`, вычисление `CONTRADICTS` и `VALIDATED_BY` поверх Neo4j.

Текущая реализация в этом дереве проще. README прямо описывает граф как `NetworkX MultiDiGraph` с JSON-персистентностью, а код действительно строит граф через [`app/graph/kg.py`](../app/graph/kg.py). Поэтому при чтении проекта полезно различать:

- **архитектурную заявку**: полноценный Neo4j/grammar/gazetteer-пайплайн;
- **фактическую реализацию**: controlled vocabulary + эвристический extractor + optional LLM + `NetworkX`-граф.

Итог: онтология здесь подана не как формальная OWL/RDF-модель, а как практичный словарь предметной области, который управляет извлечением, нормализацией, построением графа, поиском противоречий и генерацией grounded-ответов.

## Как расширять онтологию

В описанном проекте онтология **зашита в код**, поэтому расширение в лоб означает:

```text id="514vhc"
поменять app/core/ontology.py
→ перезапустить backend / build-пайплайн
→ заново прогнать extraction для документов, где нужны новые сущности/связи
→ пересобрать граф / индексы
```

Но это не единственный и не лучший способ.

## 1. Что именно придётся менять

Если сейчас есть:

```python id="bryv47"
class EntityType(Enum):
    MATERIAL = "Material"
    PROCESS = "Process"
    EQUIPMENT = "Equipment"
    PARAMETER = "Parameter"
```

и ты хочешь добавить для подземных горных работ:

```text id="shx38g"
RockMass
MineObject
GeomechanicalModel
StressStrainModel
EngineeringSolution
Risk
```

то в текущей архитектуре, скорее всего, надо править:

```text id="5himki"
app/core/ontology.py
app/nlp/extractor.py
app/graph/kg.py
frontend graph filters / colors / labels
RAG prompt, где перечислены допустимые типы
тесты / примеры
```

И да, сервер обычно надо перезапускать, потому что Python enum’ы, словари синонимов и keyword sets загружаются при старте процесса.

Но важнее другое: **старые документы сами не переосмыслятся**. Если раньше extractor не знал тип `RockMass`, то в старом графе таких узлов не появится. Нужно хотя бы выборочно переизвлечь факты.

### Как можно сделать нормально

Онтологию лучше вынести из Python-кода в конфиги:

```text id="9o7csk"
ontology/
  common.yaml
  metallurgy.yaml
  underground_mining.yaml
  geomechanics.yaml
  ecology.yaml
```

Тогда код остаётся универсальным, а предметная область меняется через данные.

Пример:

```yaml id="xw4fod"
domain: underground_mining
version: 0.1

entity_types:
  - id: MineObject
    label_ru: Горная выработка / объект ПГР
    parent: FacilityObject
    examples:
      - рудоспуск
      - штрек
      - квершлаг
      - камера
      - горизонт

  - id: RockMass
    label_ru: Горный массив
    parent: Material
    examples:
      - нарушенный массив
      - трещиноватый массив
      - удароопасный массив

  - id: GeomechanicalModel
    label_ru: Геомеханическая модель
    parent: Model
    examples:
      - модель НДС
      - модель устойчивости выработки

relations:
  - id: IN_ROCK_MASS
    from: MineObject
    to: RockMass

  - id: APPLIES_TO
    from: EngineeringSolution
    to: MineObject

  - id: MITIGATES
    from: EngineeringSolution
    to: Risk

synonyms:
  НДС: StressStrainState
  напряжённо-деформированное состояние: StressStrainState
  рудоспуск: OrePass
  ore pass: OrePass
```

Тогда расширение выглядит так:

```text id="m8sks5"
добавил YAML
→ перезагрузил ontology config
→ прогнал extraction на нужном корпусе
```

Без правки бизнес-логики.
